# 社区平台脚手架目录与接口清单

## 1. 先回答你的两个新决策

### 1.1 分包方式

你要求继续按 `controller / service / mapper` 分包，这个可以。

对你当前阶段是合理的，原因很直接：

- 项目一期是**单体应用**
- 很长时间内**流量不会大**
- 核心目标是**尽快上线 + 易维护**

所以这里不强推领域分包。当前更务实的做法是：

- **顶层按层分包**
- **每层内部再按业务文件命名**
- 公共能力单独放 `common / config / security / websocket / task`

也就是：不是微服务，也不是严格 DDD，但也不要把所有类乱堆在一个目录。

---

## 1.2 UI 组件库怎么选

### Ant Design Vue 怎么样

**能用，而且成熟。**

但如果你的目标是一个“社区交流平台”，不是后台管理系统，我的判断是：

- **Ant Design Vue**：稳定、成熟、表单和数据展示能力强，但默认视觉气质偏“企业后台”
- **Arco Design Vue**：更适合做年轻一点、视觉更轻一点的社区站，而且主题 token 能力更直观
- **TDesign Vue Next**：也可以，中文生态友好，定制空间也够

### 我的建议

#### 如果你要“前台社区站”
我建议优先：

1. **Arco Design Vue**
2. **TDesign Vue Next**
3. **Ant Design Vue**

#### 如果你要“未来前台 + 后台管理台统一一套”
可以考虑：

- **前台仍然用 Arco Design Vue**
- **后台管理系统再单独用 Ant Design Vue**

因为前台社区站和后台管理台，本来就不应该强行追求一套审美。

### 本方案默认选择

下面脚手架目录默认按 **Vue 3 + TypeScript + Arco Design Vue** 写。

如果你后面决定改成 Ant Design Vue，目录结构基本不用改，只要把：

- `src/plugins/arco.ts`

替换成：

- `src/plugins/antd.ts`

即可。

### 后端落地注意点

#### 1）Controller 不要写业务
Controller 只做：

- 参数接收
- 登录态获取
- 调 service
- 返回统一响应

#### 2）ServiceImpl 才是业务主战场
比如：

- 关注时更新关注关系 + 计数
- 点赞时更新点赞状态 + 帖子计数 + 通知
- 发消息时校验互关 + 入库 + 未读更新 + WebSocket 推送

#### 3）Mapper 只做持久化
不要把复杂业务规则塞进 XML。

### 前端落地注意点

#### 1）页面不要直接调 axios
必须统一走：

- `src/api/*.ts`

#### 2）聊天不要把所有状态塞一个组件
至少拆成：

- 线程列表
- 聊天窗口
- 消息项
- 输入框

#### 3）编辑器、上传、预览分开
这样后续切换 Markdown 组件或对象存储，不会炸全局。

## 4. 接口基础约定

### 4.1 Base URL

```text
/api/v1
```

### 4.2 认证方式

```text
Authorization: Bearer <access_token>
```

### 4.3 统一响应结构

```json
{
  "code": 0,
  "message": "OK",
  "data": {},
  "requestId": "trace-id"
}
```

### 4.4 分页结构

```json
{
  "list": [],
  "page": 1,
  "size": 20,
  "total": 100,
  "hasMore": true
}
```

### 4.5 排序字段约定

- `latest`：最新
- `hot`：最热

## 5. 接口清单

## 5.1 认证模块

### `POST /api/v1/auth/register`
- 是否登录：否
- 说明：用户注册
- 请求体：
  - `username`
  - `password`
  - `nickname`
- 返回：
  - `userId`

### `POST /api/v1/auth/login`
- 是否登录：否
- 说明：登录
- 请求体：
  - `username`
  - `password`
- 返回：
  - `accessToken`
  - `refreshToken`
  - `expiresIn`
  - `userInfo`

### `POST /api/v1/auth/refresh`
- 是否登录：否
- 说明：刷新 access token
- 请求体：
  - `refreshToken`
- 返回：
  - `accessToken`
  - `refreshToken`
  - `expiresIn`

### `POST /api/v1/auth/logout`
- 是否登录：是
- 说明：退出登录并使 refresh token 失效

### `GET /api/v1/auth/me`
- 是否登录：是
- 说明：获取当前登录用户信息

## 6. 推荐的开发顺序

按你的需求，别一上来就先做聊天。正确顺序是：

1. 用户注册 / 登录 / JWT
2. 用户主页 / 搜索 / 资料修改
3. 关注 / 粉丝
4. 帖子 / 标签 / 上传
5. 评论 / 回复 / 点赞 / 收藏
6. 通知中心
7. 浏览历史
8. 草稿箱
9. 聊天 + WebSocket

原因很简单：

- 聊天依赖登录
- 聊天依赖互关
- 通知依赖帖子 / 评论 / 关注

先做底层关系，再做消息。

## 9. 自动化回归脚本（新增）

后端目录：`community-backend/scripts`

1) 全链路 E2E（会自动启动/停止后端）

```powershell
.\scripts\run-newman-e2e.ps1
```

默认使用 `http://127.0.0.1:18080`，避免本机 `8080` 被其它进程占用导致回归误打旧服务。

2) JUnit 集成测试（Auth/Post/Comment/Notification 优先）

```powershell
.\scripts\run-integration-tests.ps1
```

说明（脚本行为）：
- 两个脚本都已加“失败即非 0 退出”，可直接接入 CI。
- `run-newman-e2e.ps1` 会在 finally 阶段自动停止后端进程；启动失败时输出：
  - `target/e2e-backend.out.log`
  - `target/e2e-backend.err.log`
- `run-newman-e2e.ps1` 会在启动前检测目标端口是否已被占用；若占用会直接失败并提示更换 `-BaseUrl`（例如 `http://127.0.0.1:18081`）。
- 建议在 `community-backend` 目录执行脚本，避免跨目录权限差异影响构建。

回归报告示例见：`docs/回归测试报告-2026-04-25.md`
